# Community OS: Talent Intelligence for START Warsaw

> Current scope update, 2026-07-12: partner-outcome collection and partner follow-up are not part of the Talent Data Room Studio report. The current build uses START-owned event evidence only. Coresignal and AI enrichment are deferred. Historical outcome ideas below remain product exploration, not current implementation requirements.

Status: concept validated in discussion, 2026-07-09. Build starts next week, after the current hackathon completes.
Origin: conversation with Aaron (EWOR) on why communities fail to land long-term sponsors, refined in a brainstorming session.

## 1. The problem

Communities and hackathon organizers struggle to get long-term monetary sponsors. That is rational from the sponsor side: they cannot trust a community on vibes. What sponsors actually buy:

- Companies: a hiring pipeline of proven, high-agency talent.
- VCs: deal flow, both current founders and pre-founders worth tracking.

Today we offer them abstract descriptions or, at best, an Excel file. Nobody else in the market gives sponsors hard, longitudinal, queryable evidence of talent quality. That is the gap.

## 2. The product in one paragraph

A system that ingests every application and participation record from our events (Luma), hackathons (Devpost), and later internal member recruitment, normalizes it into one canonical schema, enriches it (LLM signal extraction from form answers, GitHub API, Coresignal for LinkedIn), and generates two artifacts: an internal person-linked talent graph, and a partner-facing anonymized data room (self-contained HTML file) showing the talent funnel, demographics, occupation splits, talent-signal facts, and trend lines across events. "Palantir for communities". Long term: productize and sell to other communities, piloted on our own data.

## 3. Decisions made (2026-07-09)

| # | Decision | Choice |

| 1 | First slice | Partner-facing data room from hackathon/event data (external people). Internal member layer deferred to the October recruitment cycle. |
| 2 | Delivery | Static, self-contained HTML file generated by a local pipeline. No backend in v1. Optionally hosted later as a plain static URL per partner. |
| 3 | Enrichment | Form answers (LLM extraction) + GitHub API (URLs provided by applicants) + Coresignal for LinkedIn (commercial provider, not DIY scraping). |
| 4 | Age | Not asked on forms today; estimated from university years via LinkedIn education history. Shown only as aggregate buckets, labeled as estimated. |

## 4. Addressing every point raised (Q&A)

### 4.1 Voice agent ("Bordy-style" phone screener for member recruitment)
Deferred, deliberately. It is the flashiest part but the least load-bearing: it is a data source, not the product. It matters in October when member recruitment opens. Notes for then: phone-call screening needs explicit consent for recording and AI processing (stated up front in the call and in the application form); the fast-track logic ("high agency, ambitious, crazy projects -> straight to interview") is a scoring rubric we co-design; the normal application path must always exist as an equal option. All agent-collected data flows into the same canonical schema, which is why the schema comes first. ElevenLabs Agents is the likely build path (skill already available in our tooling).

### 4.2 Category taxonomy ("20% big tech, 30% founders, 35% built something impressive")
Correct instinct, and it must be a fixed, versioned taxonomy, not free-form LLM labels, otherwise percentages are not comparable across events and trend lines become meaningless. We co-design the taxonomy before the first pipeline run: occupation category (student / founder / big tech / startup employee / researcher / other), builder-signal tiers (shipped something impressive, with a rubric for "impressive"), standout facts. The LLM classifies into the fixed taxonomy; a manual review pass on low-confidence classifications keeps quality.

### 4.3 Karpathy LLM-wiki (living, updatable knowledge base per person)
Right concept for the internal layer, October phase. Every person gets a document that accumulates: application answers, enrichment snapshots, event participation, project outcomes, manual notes pushed via a chat (or later caller) pipeline. Executive summaries are regenerated from the document, so the partner-facing aggregates and the internal view always derive from the same substrate. In v1 we implement the substrate (canonical store with per-person records and update timestamps); the wiki UX comes later.

### 4.4 Funnel visualization (horizontal tunnel, dots = people)
Kept, with the privacy-safe variant: dots render real volumes and flow between stages, but are texture, not links to individuals. Clicking a stage (or dot) opens stage-level aggregates. The internal version (October) can make dots clickable to person summaries, because that view is access-controlled and person-linked by design.

### 4.5 Funnel stages
Hackathon/event funnel v1: registered -> accepted -> checked in / participated -> submitted a project -> outcome (hired / invested / joined community). Member funnel (October): applied -> interviewed -> accepted -> active -> alumni (exit to startup, job, etc.). Alumni is a first-class stage: it is where the strongest sponsor evidence lives ("people who went through us go on to X").

### 4.6 Cross-hackathon trends
Per-event data room plus a combined view: application volume per event over time, talent-signal composition over time (taxonomy percentages), acceptance rates, submission rates. This is why the canonical store is append-only and event-scoped: trends come free once two events are ingested. This is also the moat: any single report can be faked with effort, a longitudinal record cannot.

### 4.7 Outcome tracking (partners hiring/investing)
Asking partners to self-report will not happen reliably. Control point moved to our side: every partner-to-person introduction goes through us and gets logged (a lightweight intro log: who, to whom, when, context). Partner confirmation then becomes a follow-up on a known intro, not an open-ended reporting request. Self-reported outcomes from participants (exit surveys, alumni updates) are the second source.

### 4.8 Unification layer
Yes, and it is the core engineering asset. Canonical schema (person, event, application, participation, enrichment, outcome) plus per-source adapters (Luma CSV, Devpost CSV, custom forms, future: voice agent transcripts). Field mapping is config-driven so a new source or a changed export format is a mapping file, not a code rewrite. Manual step stays exactly as you wanted: drop exports into a folder, run the pipeline.

### 4.9 PostHog in a static HTML file
Works. posthog-js is a plain script snippet; it needs only that the viewer is online. EU cloud host (eu.i.posthog.com) for GDPR consistency. Custom events per section (funnel viewed, demographics viewed, highlights viewed, time on section) give exactly the "what do partners care about without telling us" signal. localStorage persistence handles file:// opens; each generated file is stamped with partner and event properties so events are attributable without identifying the viewer. Footer disclosure line required; silent fingerprinting of the exact people whose trust we are building would be self-defeating.

### 4.10 Coresignal for LinkedIn
Accepted as the enrichment provider (reliable, commercial, used widely by VC/recruiting tooling). Legal note: buying from a broker improves practical reliability but we remain the data controller for the enrichment. GDPR Art. 14 requires notifying people when data about them is collected from third parties; the clean way is a line in the application form privacy notice ("we may enrich your application with publicly available professional data"). Research doc has the details and enforcement precedents.

### 4.11 Anonymization ("its anonymized" for external view)
Aggregates-only plus a k-anonymity floor: no statistic or highlight fact is shown if the underlying bucket has fewer than k people (k=5 as default). Highlight facts ("chess grandmaster at 12, first million at 20") are the riskiest element: in a 30-person cohort they are identifying. Policy: highlight facts are either (a) consented explicitly by the person, or (b) blurred until unlinkable, and never combined with other quasi-identifiers on the same card. This policy is enforced in the pipeline (the renderer refuses to emit sub-threshold aggregates), not by manual discipline.

### 4.12 Paid continuous partner access (later)
Logged as a phase 3+ product: partners pay for a living, continuously updated view, with consented person-level visibility (people opt in to being visible to partners, with contact details). That is also the natural monetization for the productized version. Requires the hosted architecture and a real consent flow; explicitly out of scope until the static data room proves demand. The static file is, among other things, the cheapest possible sales test of that product.

### 4.13 "Community OS" productization
Sequencing: prove on our own data (pilot = our hackathons), then generalize. What makes it sellable to other communities is exactly what we need anyway: config-driven source adapters, fixed taxonomy, one-command generation, embedded analytics. Avoid premature generalization: build for START Warsaw first, extract the product later. The competitive research (docs/research/competitive-landscape.md) covers whether anyone already does this.

### 4.14 "Maybe you find a better approach"
Two changes vs the original idea, both already agreed: (1) slice order, data room first, voice agent last; (2) partner view is aggregates-only rather than clickable anonymized individuals. One addition: the intro log (4.7) replaces hoping partners report outcomes. Otherwise the original architecture (one canonical store, everything derived from it, unification layer, static export) survives scrutiny and matches how I would build it.

## 5. Risks and open questions

- Devpost data access: exact export fields and whether we get emails/profiles of all registrants vs only submitters. Research doc + verify against a real export next week.
- Luma API tier: API access may require a paid plan; CSV export works regardless.
- Consent retrofit: data already collected (past events) was likely collected without a partner-stats consent line. Aggregate-only statistics over it are defensible (statistical purpose, anonymized output), but highlight facts from past-event people need consent or heavy blurring. All future forms get the consent lines from day one.
- Taxonomy design: needs a working session (team) before first pipeline run.
- Small-cohort math: with 20-50 people, several demographic cuts will fall under k=5 and render as "not enough data". That is correct behavior, not a bug.
