# Residual-risk register

| ID | Residual risk | Current control | Decision owner | Status before live release |
|---|---|---|---|---|
| RR-001 | Publicly available profile data remains personal data and may be incompatible with the intended purpose or provider terms | Applicant-source evidence, purpose/allowed-use inventory, hard source gates | the release owner plus qualified privacy/legal reviewer | Open, blocking until reviewed |
| RR-002 | Coresignal contract, source authorization, regional processing, and retention may not match this use | Separate post-notice bound gate, managed token, disabled default | the release owner plus qualified privacy/legal reviewer | Open, blocking Coresignal |
| RR-003 | AI processor DPA, retention/training mode, region, security settings, and field allowlist may be inadequate | OpenAI Responses adapter is hard-gated, sends only pseudonymous structured codes with `store: false`, and requires the exact processor record plus a managed key | the release owner plus qualified reviewer | Open, blocking external AI until the actual project settings are verified |
| RR-004 | An excluded applicant may use a different email in an attendance export, preventing deterministic attribution of that attendance row | Application IDs are removed across applicant-bound stages; exact normalized emails remove linked attendance rows; any ambiguous linkage must remain in human review and block publication | START privacy lead | Open, blocking an affected release until ambiguous links are resolved |
| RR-005 | A malformed, duplicate, or unknown subject pseudonym could cause incomplete exclusion propagation | The exclusion plan fails closed, binds the known applicant population, invalidates all existing local derivatives, and stores only count plus deterministic set hash | Release reviewer | Mitigated in code; fixture-tested and still required in every release QA |
| RR-006 | Local cleanup now removes expired uploads, caches, stages, private reports, review bindings, and public staging, but pseudonymous audit history and backups can remain stored | Daily `privacy-cleanup` plus deployment-specific audit and backup lifecycle jobs | START privacy lead plus infrastructure owner | Open, blocking deployment until external lifecycle jobs are verified |
| RR-007 | Coresignal retention may still be longer than necessary even within the machine maximum | Gate is limited to 1 to 30 days; approval records the exact period and qualified review must select the shortest justified value | the release owner plus qualified privacy/legal reviewer | Mitigated ceiling; live value still requires approval |
| RR-008 | Multiple replicas with different operator roots would not share the cleanup/operator filesystem lock | Cleanup and operator mutations use `<root>/.operator-mutation.lock`; deployment must mount one protected root in every replica | START infrastructure owner | Open, blocking multi-replica rollout until topology is verified |
| RR-009 | Repeated aggregate releases may enable differencing or reidentification | Minimum-cell and complementary suppression; publication manifest and external release ledger required | START privacy lead | Open |
| RR-010 | Production operator storage may lack encryption, tested backup/restore, deletion propagation, or key rotation | Protected file modes and deployment contract | START infrastructure owner | Open, blocking deployment |
| RR-011 | Authenticated colleague account or proxy compromise can expose raw data | Identity proxy, exact allowlist, CSRF, mutation lock, audit, least privilege | START infrastructure owner | Open |
| RR-012 | Storage-specific deletion can fail after partial external side effects | Fail-closed cleanup, deletion counts, pseudonymous audit; alert on command failure | START privacy lead | Monitor and test |
| RR-013 | Taxonomy labels can be wrong, biased, or consequential despite confidence | Unknown states, evidence references, mandatory human review | START talent evidence reviewer | Open review queue blocks release |
| RR-014 | PDF extraction/scanning may miss visual or embedded disclosure | Same-evidence generation plus visual and text parity checks | Release reviewer | Required every release |
| RR-015 | `public-staging` may be mistaken for a completed publication | Staging manifest says analytics disabled; deployment hash check and external publication receipt are separate | Release owner | Open, publication not automated |
| RR-016 | Analytics may drift, collect network identifiers, or add unapproved dimensions | Separate hash-bound, short-lived approval; project-level IP-capture confirmation; allowlisted EU origin, events, and properties; per-load identity; no person profile or storage; CSP and final manifest verification | Product research owner | Enable only after exact staging; verify the remote IP setting and post-deploy traffic |

No row in this register is accepted merely because code exists. the release owner or the named accountable owner must explicitly accept, mitigate, or keep the release blocked. `Safe to publish` is a local machine state, not a legal opinion or a claim of zero residual risk.
