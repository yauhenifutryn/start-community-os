# Talent pipeline data processing and retention matrix

Status: release-readiness control document, not a legal-compliance determination.

Accountable owner: `privacy_lead`. the release owner remains the separate processing, publication, and Coresignal approval authority. A qualified privacy or legal reviewer must confirm source authorization, legal basis, notices, processor terms, regional/security settings, and retention before live enrichment.

| Data class | Source | Purpose | Minimum fields | Machine-allowed uses | Protected location | Enforced or configured retention | Automated deletion | Public output |
|---|---|---|---|---|---|---|---|---|
| Raw application export | Applicant-provided Luma export | Reconcile the applicant cohort and source evidence | Stable source row, identity links, structured answers needed by the taxonomy | `aggregate` | `<root>/protected/uploads/` | Absolute `privacy_operations.retention_deadline` must be in the future | `privacy-cleanup` deletes the expired upload, clears its source slot, and deletes all local derivatives | Never |
| Attendance supplement | Organizer export | Reconcile accepted and on-site totals | Stable source row, approval state, check-in state | `aggregate` | `<root>/protected/uploads/` | Absolute `privacy_operations.retention_deadline` must be in the future | `privacy-cleanup` deletes the expired upload, clears its source slot, and deletes all local derivatives | Never |
| Track preferences and submissions | Applicant and Devpost exports | Resolve teams, projects, and submission evidence | Team link fields, track, project evidence, applicant-supplied artifact links | `aggregate` | `<root>/protected/uploads/` | Absolute `privacy_operations.retention_deadline` must be in the future | `privacy-cleanup` deletes the expired upload, clears its source slot, and deletes all local derivatives | Never |
| GitHub response and cache | Applicant-supplied GitHub identifier through an authorized source | Observe bounded public builder evidence | Repository count, account age, last public update, pseudonymous evidence reference | `classify` | `<root>/protected/stages/github.json` and protected cache | Gate value, 1 to 30 days | `privacy-cleanup` physically deletes expired cache/raw stage data and invalidates dependent outputs | Aggregate cells only after reviewed classification |
| Applicant public page and cache | Applicant-supplied HTTPS URL | Observe bounded portfolio evidence | At most 4,000 visible-text characters and a pseudonymous evidence reference | `classify` | `<root>/protected/stages/public_pages.json` and protected cache | Gate value, 1 to 14 days | `privacy-cleanup` physically deletes expired cache/raw stage data and invalidates dependent outputs | Aggregate cells only after reviewed classification |

| Semantic classifier input/output | Pseudonymous structured evidence sent through an approved OpenAI Responses project | Propose versioned taxonomy labels | `subject_ref`, code-only `signals`, pseudonymous `evidence_refs`, labels, confidence, versions, review state | `aggregate` | `<root>/protected/stages/classification.json`, protected cache, and review store | 30 days for cache/stage output; processor-side mode must be separately recorded and verified | `privacy-cleanup` physically deletes expired cache/stage data and invalidates dependent outputs | Reviewed aggregate cells only |
| Review and operator audit records | Authenticated operator actions | Record who accessed, exported, approved, rejected, corrected, or ran a stage | HMAC-pseudonymous colleague, pseudonymous subject/case, code-only action/reason, time, version | Audit and incident response only | Protected operator state, review repository, `operator-access-audit.jsonl`, and external audit sink | Deployment retention schedule, subject to qualified review | Not covered by `privacy-cleanup` | Never |
| Privacy Operations record | Approval bundle plus derived inventory | Track source, purpose, owner, deadline, rights state, allowed use, and release state | Code-only owner/status values, timestamps, version/hash references | Release evaluation and audit | `<root>/protected/privacy-operations.json` | Deployment retention schedule | Not covered by `privacy-cleanup` | Never |
| Aggregate contracts and report preview | Derived, reviewed cohort evidence | Render and review the partner report | Counts, denominators, privacy state, coverage, limitations | `publish` only after approval | `<root>/protected/release/` | Absolute bundle deadline plus a separately approved rollback window | Not covered by `privacy-cleanup` | Only through explicit public staging |
| Public staging | Hash-bound approved aggregate release | Isolate the host-independent content allowlist | Approved `index.html` and `partner-talent-brief.pdf` plus `publication-manifest.json` | Derive or deploy only that approved release | `<root>/public-staging/` | Approved publication lifetime and rollback policy | Deployment/storage-specific removal | Yes, after release gate and deployment review |
| Deployment staging | Separately approved public staging plus analytics policy | Bind Vercel response headers and fixed interaction measurement to exact approved bytes | `vercel.json`, analytics-enabled `index.html`, approved PDF, final manifest | Preview deployment of that exact bundle only | `<root>/deployment-staging/` | Approved publication lifetime and rollback policy | Deployment/storage-specific removal | Yes, only after preview verification and production approval |
| Privacy-limited analytics | Deployed public partner report | Learn which aggregate cohorts, metrics, and overlap regions support partner decisions | Report version hash, cohort key, metric key, overlap region | Product research only | Approved EU PostHog project plus protected activation artifact | Shortest configured project retention | PostHog project TTL and deletion controls | No participant data |

Machine controls live in `community_os/controlled_release.py`, `community_os/privacy_operations.py`, `community_os/enrichment/gates.py`, `community_os/enrichment/classification.py`, `community_os/publication.py`, and `community_os/postpublication_analytics.py`. Missing rights or approval records fail closed. Valid subject exclusions propagate across every applicant-bound and public surface; malformed, duplicate, or unknown pseudonyms fail closed. Unused stored data remains exposure and responsibility: raw sources, provider/classification caches, stages, private reports, review bindings, and public staging are deleted automatically at the relevant source deadline. Pseudonymous audit history and backups still require separately configured lifecycle jobs.
